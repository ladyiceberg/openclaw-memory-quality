## 一、记忆系统问题清单

### 问题 1：信号积累断路

**现象**：我不主动走 memory_search → 对话内容不会变成信号

**根因**：
- 我回答问题时用 exec/grep 直接读文件更快，不产生信号
- memory_search 需要手动调用，我没有这个习惯
- 系统没有强制我每次回答前必须搜一次

**影响**：所有对话内容都无法进入 recall store

---

### 问题 2：日志漏写

**现象**：memory/YYYY-MM-DD.md 经常是空的或不完整

**根因**：
- AGENTS.md 说要读/写每日日志，但没有自动化这个动作
- 对话结束了，我不会自动检查"今天日志写了没"

**影响**：记忆系统失去最重要的原始数据来源

---

### 问题 3：新文件索引真空

**现象**：创建了 2026-04-09.md，但 memory_search 搜不到

**根因**：
- 新文件不会被自动索引
- 需要手动 `openclaw memory index`
- 索引 Rebuild 之后才能搜

**影响**：新建的日志文件在索引更新前是完全"沉默"的

---

### 问题 4：搜索结果匹配不稳定

**现象**："羽毛球"明明在文件里，但 memory_search 返回 0 结果

**根因**：
- embedding 匹配有阈值，低于阈值不返回
- chunk 切分可能把相关内容分开了
- 0 结果的搜索不写入 events.jsonl（不产生信号）

**影响**：某些内容永远无法变成信号

**典型案例（2026-04-09 晚）：**
- 文件 `memory/2026-04-08.md` 中存在羽毛球相关内容：
  - Line 42: `晚上要打羽毛球（20:00开始）`
  - Line 50: `晚上要打羽毛球（8点开始，没穿合适的运动鞋）`
  - Line 64: `打羽毛球回来了，没怎么上场，观看为主`
- 用 `grep` 能找到，但 `memory_search("羽毛球")` 返回 0 结果
- 尝试了多种搜索词组合均失败：
  - `羽毛球` → 0 结果
  - `羽毛球 运动 打球` → 0 结果
  - `晚上 运动` → 0 结果
- 对比：同一文件中 "种花" 能搜到，但 "羽毛球" 搜不到
- 两者在文件中的出现频率相似，但 embedding 匹配差异巨大

**补充验证（2026-04-09 23:20）：**
- 执行 `openclaw memory index --force` 重建索引后，问题依旧
- `Vector: ready` + `3/3 files · 9 chunks` → **词已向量化**
- 但搜索 `project` 能搜到，搜索 `重要 待办` 返回 0
- **结论**：词已向量化，问题在于 **chunk 切分或 embedding 相似度阈值**
- 可能原因：文件被切成 9 个 chunks 后，"羽毛球"落在某 chunk 内，该 chunk 的向量与 query 相似度低于返回阈值

---

### 问题 5：Dreaming 不 survive 重启

**现象**：Gateway 重启后，dreaming cron job 消失

**根因**：
- cron job 注册信息存在内存或临时文件，重启丢失
- 系统没有自动重新注册机制

**影响**：记忆晋升完全停止

---

### 问题 6：格式不统一

**现象**：我主动写的 vs dreaming 晋升的，格式完全不同

**根因**：
- 我用 bullet 格式直接写 MEMORY.md
- dreaming 用结构化格式（分数+标签+片段）
- 两套逻辑，没有统一规范

---

### 问题 7：上下文看不到日志文件

**现象**：我每次醒来，memory/YYYY-MM-DD.md 不在我的上下文里

**根因**：
- Project Context 只加载 AGENTS.md / SOUL.md / MEMORY.md
- 每日日志不在预加载范围
- AGENTS.md 说要读但实际没自动读

**影响**：我每次回答是"空白的"，要靠你说才知道上下文

---

### 问题 8：Chunk 语义分散导致 maxScore 天花板低（2026-04-13 新发现）

**现象**：recall store 里多条记录的 maxScore 卡在 0.4-0.57，dreaming 的 0.8 阈值完全够不着

**根因**：日志按时间顺序写，但 embedding 按语义 chunk 索引。当一个 chunk 包含两件以上不相关的事，语义散掉，任何单一 query 都只能匹配其中一部分。

**典型案例**：
- chunk 74:96 同时包含"教训"和"待办列表"，maxScore=0.401
- chunk 74:97 只有"教训+讨论"，语义集中，maxScore=0.487
- 同文件相邻两条，差异来自 chunk 内容本身，不是模型问题

**解法**：日志按语义分区块写，每个区块语义独立。

---

### 问题 9：Promote 阈值无法调整（2026-04-13 新发现）

**现象**：dreaming 的 minScore=0.8 是内部硬编码，无法通过配置调整

**根因**：阈值是 memory-core 内部参数，没有暴露给用户配置

**影响**：即使 recallCount 和 uniqueQueries 够，maxScore 不够就永远过不了

## 五、日志格式规范（2026-04-13 新增）

**核心原则：按语义分区块写，不要按时间流水账**

**Before（问题写法）**：
```markdown
### 傍晚（18:34 开始）
- **18:43**: 晚上要打羽毛球（20:00开始）
- **18:52**: 切换网络，Gateway 重启后测试搜索，仍失败
- **19:04**: 晚上要打羽毛球（8点开始，没穿合适的运动鞋）
- **19:36**: 聊创业项目
```

**After（正确写法）**：
```markdown
### 傍晚（18:34 开始）

#### 运动
- 晚上要打羽毛球（20:00开始）

#### 创业
- 想重启，跟记忆相关

#### 网络配置
- Google Gemini API 配置完成，发现 MacBook Air 无法访问外网
```

**为什么这样有用**：每个 subsection 的语义是独立的，chunk 不会被旁边不相关的内容稀释。

## 二、记忆质量管理系统设计方案

### 模块 1：信号积累层（最核心）

**目标**：让每条对话内容自动变成 recall 信号

**方案 A：对话拦截写入器**
- hook 到我每次回答的流程
- 自动提取关键词 → 调用 memory_search → 写入 recall store
- 用户不需要提醒我

**方案 B：定期批量转换**
- 每小时或每次 heartbeat 时
- 扫描 recent session transcript
- 提取关键段落 → memory_search → 写入

**推荐方案 A**，因为信号积累需要及时性

---

### 模块 2：日志维护层

**目标**：memory/YYYY-MM-DD.md 自动创建、自动更新

**方案**：
- 对话开始前：检查今天日志是否存在，不存在则创建
- 对话结束后：自动把重要片段追加到日志
- 格式统一：`## YYYY-MM-DD` + 时间戳条目

**触发时机**：
- 每次 heartbeat 时检查
- 或者每次新对话开始时检查

---

### 模块 3：索引健康管理

**目标**：新文件创建后自动重建索引

**触发条件**：
- 检测到新的 .md 文件 → 自动触发 `openclaw memory index`
- 或者定时检查（每小时一次）

**健康检查**：
- 定期报告索引覆盖率
- 如果文件数 ≠ 索引里的文件数 → 报警

---

### 模块 4：Dreaming 可靠性保障

**目标**：确保 dreaming cron job 始终存在、重启后自动恢复

**方案**：
- ** watchdog**：每次 heartbeat 检查 dreaming cron 是否存在
- 不存在 → 自动重新注册
- 记录历史运行记录，失败告警

**状态面板**：
- 下次运行时间
- 最近一次运行结果
- 晋升了哪些条目

---

### 模块 5：搜索质量监控

**目标**：发现并修复 memory_search 匹配问题

**方案**：
- 定期用已知存在的关键词测试搜索（如"种花"）
- 对比 grep 结果 vs memory_search 结果
- 不一致 → 触发 reindex + 报告问题

**可配置的搜索阈值**：
- 降低相似度阈值，让更多内容能被匹配
- 或者改进 chunk 策略

---

### 模块 6：用户可观测性面板（Dashboard）

**目标**：让用户能看到整个记忆系统状态，并干预

**建议的功能**：

| 功能 | 说明 |
|------|------|
| **recall store 浏览器** | 查看当前所有信号、recallCount、分数 |
| **待晋升列表** | 显示哪些候选即将被晋升 |
| **手动晋升/降级** | 用户可以主动晋升或删除某条信号 |
| **Dreaming 日志** | 今晚 3 AM 发生了什么，用户能看到 |
| **信号来源追踪** | 某条记忆是被哪次对话触发写入的 |
| **记忆健康分** | 整体评分，类比"系统健康度" |

---

### 模块 7：上下文预加载优化

**目标**：让我每次醒来能看到最近对话日志

**方案**：
- 优化 AGENTS.md 的 startup 行为
- 确保 memory/YYYY-MM-DD.md 被加载到 Project Context
- 或者在 HEARTBEAT.md 里写明"每次启动必须读今日+昨日日志"

## 三、优先级建议

**第一优先级（必做）：**
1. 模块 1（信号积累）— 解决最核心的断路问题

6. 模块 5（搜索质量）— 提升匹配率

## 六、Recall Store 优化实战手册（2026-04-14 新增）

### 核心发现

1. **词序调整是突破瓶颈的王道**
   - 同一个词组合，词序不同可以产生截然不同的分数（0.775→0.797→0.800）
   - "重要发现"放句首对某些chunk突破瓶颈极其关键

2. **中英混合词效果最好**
   - 纯英文词（如"Gateway restart"）搜不到结果
   - 中文+英文混合（如"Gateway 重启"）效果最好

3. **精准术语比泛词更有效**
   - "Dreaming cron job delivery target 格式问题" 比 "Dreaming cron" 精准得多

4. **组合大小适中**
   - 6-8个词的组合通常最优
   - 太短（3-5词）信号弱，太长（9+词）引入噪音

### 加分词（提升score）
- 目标chunk独有的术语
- 中英混合词
- 精准描述性短语
- 对大小写敏感（大写"OpenClaw"≠小写"openclaw"）

### 降分词（降低score，必须避开！）
| 降分词 | 影响 | 原因 |
|--------|------|------|
| 信号池 | -0.303 | 同时出现在多个chunk，拉向其他chunk |
| 重要发现 | -0.303 | 高频词，多个chunk共享 |
| 三阶段 | -0.319 | 拉向Dreaming三阶段相关chunk |
| 成功 | -0.291 | 太泛，引入噪音 |
| 日志 | -0.276 | 多处出现，拉向其他chunk |
| 系统 | 大幅降分 | 太泛，拉向所有讨论系统的chunk |
| 创业 | 大幅降分 | 拉向其他创业相关chunk |
| 重要教训 | 大幅降分 | 拉向多个有"重要教训"的chunk |
| 数字/时间（如11、39、5月） | 基本无效 | 搜不到或降分 |

### 典型案例

**74:97 chunk（差0.005触顶）**
- 内容：待办清单+记忆系统讨论+漂头发+草莓音乐节+种花+创业+晋东南游学 = 7-8个主题
- 问题：内容太杂，语义上限卡死
- 最佳组合："short-time-recall.json 假文件 重要待办 草莓音乐节 漂头发 种花 cron Dreaming" = 0.795
- 结论：chunk内容太杂时，即使最优query也难以突破0.8

**09:1:35 chunk（从0.559→0.755）**
- 内容：Dreaming cron job调试+记忆系统测试
- 最佳组合："Dreaming cron job delivery target 格式问题 Gateway 重启 openclaw memory promote cron add recall store bug memory_search 四层结构 上午" = 0.755
- 关键：找到这个chunk独有的精准术语组合

### chunk内容太杂的解决思路
- 日志按语义分区块写（见第五章格式规范）
- 每个subsection语义独立，chunk不会稀释
- 等待Dreaming自动整合或系统更新

### 搜索优化Checklist
- [ ] 优先找目标chunk独有的术语
- [ ] 避开降分词列表
- [ ] 尝试中英混合词
- [ ] 词序调整（"重要发现"放句首）
- [ ] 组合大小控制在6-8词
- [ ] 大小写敏感（OpenClaw≠openclaw）
